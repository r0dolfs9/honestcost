Build guide · v1.0 · 2026-05-25

Audit the math before
you sell anything.

A reusable specification for the formula audit tool — built once for HonestCost, reusable for the next calculator. This document is the plan. The output is a single HTML file: swipe through every formula, accept-deny-flag each, end on a transparent report you can hand to a dealer, a journalist, or a future you.

01 / the goalWhat "audit complete" looks like.

A trustworthy calc engine is one where every atomic formula has been individually defended in plain language by a human who could have rejected it — and didn't. That's what we're building.

The audit is done when:

  • Every atomic formula in the engine has a card.
  • Every card has been reviewed: accepted, rejected, or flagged with a note.
  • The summary report exports as a single text/markdown blob — the "audit trail" you can paste anywhere.
  • Rejected formulas have an action item attached (rewrite, replace, remove).
  • You'd be comfortable showing the report to a skeptical engineer.

What this is NOT

This is not a unit test runner. Unit tests prove the code does what the code says. The audit asks the harder question: does what the code says match reality? A formula can be perfectly implemented and still be wrong.

02 / the artifactOne file. Swipe through. Done.

The output is a single self-contained HTML file: formula-audit.html. Open it, swipe through the cards, finish. State persists in localStorage so closing the tab doesn't lose progress. At the end you get a summary screen with one-click export.

The driver loop, conceptually

// formula-audit.html — pseudocode
const FORMULAS = [/* one entry per atomic formula — see schema */];
const state = loadFromLocalStorage() || initialState();

function renderCard(index){
  const formula = FORMULAS[index];
  const verdict = state.verdicts[formula.id];   // 'accept' | 'reject' | 'flag' | null
  paintCard(formula, verdict);
  paintProgress(index, FORMULAS.length, state);
}

function onVerdict(id, verdict, note){
  state.verdicts[id] = { verdict, note, ts: Date.now() };
  saveToLocalStorage(state);
  advance();
}

function onComplete(){
  renderSummary(FORMULAS, state);   // last screen
}

That's the whole architecture. ~200 lines of JS, ~150 lines of CSS, plus the FORMULAS data array — which is the part that changes per calc.

04 / the data shapeOne schema, every calc.

This is the contract. Same schema for HonestCost, same schema for your next calculator. Fill it once per formula and the audit tool renders without changes.

FieldTypeRequiredWhat it does in the card
idstring · uniqueyesKey for state persistence. Use the actual function name: ten, kasko.
namestringyesTitle of the card. Plain language, not the function signature.
categorystringyesPill in the corner. tax · finance · insurance · operating · depreciation · fuel · custom.
shortDescriptionstring · 1 sentenceyesSubtitle under the name. One line that frames what the formula does.
whyItExistsstring · 2–3 sentencesyesThe defense of the formula. Why is this even in the calc? What would break without it?
formulastring · math notationyesThe math, not the JS. One line if possible; multi-line OK with proper indentation.
formulaCodestring · JSoptActual code from the source file. Useful when implementation differs subtly from math.
inputsarray<{name, unit, source}>yesInputs grid. source = where this value comes from in practice (form field? CAR_DB? user choice?).
output{unit}yesWhat comes out and in what unit. Bottom row of the inputs grid.
assumptionsarray<string>optBullet list of explicit assumptions. Anything you'd have to defend in an interview.
edgeCasesarray<{case, behavior, verified}>yesMin 3 verified edge cases per formula. Forces you to have actually tested it.
sourcesarray<{label, url, date}>yesWhere the numbers came from. Min 1 per formula. "My memory" doesn't count.
riskLevel'low' | 'med' | 'high'yesHow bad if we're wrong? Pill in the corner. Drives review priority.
dependsOnarray<formula.id>optOther formulas this one consumes. Lets the summary view show the graph.

Why this schema, and not less

Two fields earn their place: whyItExists forces you to defend the formula in plain language (the moment you can't, you've found a problem). edgeCases · min 3 forces you to have actually run the formula, not just looked at the code. Everything else flows from those two.

05 / card anatomyTop-down, decision-density-ordered.

A card reads from "I have no idea what this is" to "I'm ready to accept" in 8 seconds for a low-risk formula, 90 seconds for a high-risk one. The vertical order matters.

  1. Category pill + risk pill (top corners) — instant frame. Tax vs depreciation vs insurance + low/med/high. Drives how much attention to give.
  2. Plain-language name + 1-line description — "what is this thing" answered before any math appears.
  3. Why it exists — the defense. Most important field. If a reviewer disagrees here, the rest is moot.
  4. Formula in math notation — one line ideally. The shape of the math, not the code.
  5. Inputs grid — each input with unit and source. Tells you what the formula needs and where each input is sourced in the live app.
  6. Edge cases verified — 3+ pass/fail rows. Proves the formula was actually run.
  7. Sources — links + date. The "why should I believe these numbers" answer.
  8. Sticky actions — Reject / Flag / Note / Accept, big touch targets, keyboard-accessible.

Risk-level guidance

LevelDefinitionExampleReview depth
lowWrong by ±10% changes nothing material in the output.TEN tax (≤€520/yr in a €15k+ total)Quick read · 8s
medWrong by ±10% noticeably shifts the verdict.Service cost, tyre costRead sources · 30s

06 / flow & uxHow the swipe works.

Navigation rules

  • Right swipe / A key / accept button → mark accept, advance to next card.
  • Left swipe / R key / reject button → mark reject, prompt for "what's wrong?" note, advance.
  • Up swipe / F key / flag button → mark flag (needs more info / parked), advance.
  • N key / pencil button → add note without changing verdict.
  • ← → keys → navigate without verdict (review previous card). Verdict still persists.
  • Esc → open summary view at any time.

Persistence

State is checkpointed every action. Closing the tab loses zero. Returning resumes on the first un-verdicted card. Add a per-audit ID (honestcost) so the same tool can audit a different calc without state collision.

Animations — keep them honest

  • Card translates off-screen on action (~350ms, eased) — gives the gesture weight.
  • Behind-card scales up and forward — the "next one's ready" affordance.
  • Toast in the corner confirms save — no popup, no modal, no friction.
  • Progress dots fill as you go — the gamification that earns the next 30 minutes.

Export options on the summary screen

  • Copy as Markdown — for pasting into a doc / GitHub README / dealer FAQ.
  • Download as JSON — the raw state.verdicts blob. Re-importable to resume.
  • Print to PDF — single-page audit report you can attach to the dealer pitch.
  • Generate "Kā mēs aprēķinām" snippet — auto-build the HTML trust-section from accepted formulas. Connects directly to the integration snippet.

The downstream payoff

The "Kā mēs aprēķinām" trust section on the live site is currently hand-written. Once the audit is built, it can be auto-generated from the verdicts. Each accepted formula becomes one trust-section card with its whyItExists as body and sources as footer. Audit and dealer-credibility surface = same artifact.

08 / reuseFive steps to audit the next calc.

The whole point of the schema is that the tool doesn't change between calcs. The data does. Here's exactly what changes when you point the audit at a different calculator.

  1. Copy formula-audit.html next to the calc. The audit file is self-contained. No build step, no dependencies, no backend. Drop it in the same folder as the calc's index.html. 5 min
  2. Update the AUDIT_ID constant. One line at the top: const AUDIT_ID = 'honestcost' → 'mortgagecalc' (or whatever). This isolates the localStorage so the two audits don't collide. 1 min
  3. List the atomic formulas in the calc. Open the calc's source. Every function that returns a number and isn't just plumbing (DOM reads, formatting) is a formula. Aim for 5–15 entries. For HonestCost: 9. For a simpler mortgage calc: probably 4–6. 15 min
  4. Fill in the FORMULAS array using the schema. One entry per formula. Sit down with the source code open. Hardest field: whyItExists — if you struggle to write it, the formula might not belong in the calc. Easiest: formulaCode — copy-paste from source. 10–20 min per formula
  5. Open the file. Audit. Export. Sit down, do all the cards in one or two sessions, paste the markdown export into the calc's docs. Done. 30–90 min total

What you DON'T do for a new calc

You don't touch the HTML structure. You don't touch the CSS. You don't touch the swipe / keyboard / persistence logic. The only file content that changes is the FORMULAS array and the AUDIT_ID string. That's the reusability test — if you find yourself editing rendering code, the schema is wrong.

09 / honestcost backlogThe 9 cards we're about to fill.

Sorted high-risk-first. High-risk formulas can flip the comparison verdict, so they earn the deepest review and the most-skeptical sources.

# Formula What it does Why it's hard Risk
01 residual(price, cat, yrs) Compound depreciation per segment 5 segment rates × 10 years = 50 numbers, all from market scrapes. Wrong by 5% = comparison flips. High
02 leasing(p, dp, apr, n, res) PMT with residual discount Standard formula but easy to misread residual handling. APR sensitivity is huge. High
03 kasko(price, rate, cat, yr) KASKO on declining residual Rate × residual is right; tier rates (1.5–3.5%) are scraped, not quoted. EV penalty is folklore. High
04 repairBuf(level, war, cat, yr) Post-warranty maintenance buffer + 10% YoY The 10% escalation is by feel. The –20% Latvia discount needs more than one source. Med
05 fuelAnnual(car, km) ICE / EV / PHEV branches with charging mix EV home charging % is a single user-set knob; 66% default is current data but volatile. Med
06 service(km, yr, interval, cost) Visits = floor(km × yr / interval) Time-floor (1 visit per 2 years) was added to fix low-km bug. Check it's wired in calcAll. Med
07 autoOCTA(x) OCTA estimate by kW 5 tier costs; "experienced Riga driver" baseline is implicit and disclaimed. Junior drivers way off. Low
08 tyres (TYRE_ANNUAL_COST) Flat €/yr by tyre class Plain lookup. Easy to defend; tyre prices LV are pinned to Continental/Michelin sheets. Low
09 ten(co2) WLTP CO₂ → annual road tax Plain lookup. 14-bracket table is law. Hard to be wrong if the table is right. Low

Audit order rationale

We review high-risk first because the worst-case is finding a flaw on card 9 and realizing it invalidates cards 1–8. High-risk-first means the cheapest discoveries happen earliest. If residual turns out to be wrong, you stop the audit, fix the data, and restart — having spent 5 minutes, not 45.

10 / exit criteriaWhen to stop and ship.

The audit is done — and the trust section, the dealer pitch, and the journalism pitch are all unlocked — when:

  • All 9 HonestCost cards have a verdict.
  • No high-risk formula is in reject state without an action item.
  • Every flag has a 1-line note explaining what's needed to resolve.
  • The summary markdown export is saved to FORMULA_AUDIT_REPORT.md in the repo root.
  • The auto-generated "Kā mēs aprēķinām" snippet has replaced the hand-written one in index.html.

What you do after audit:

  1. Fix every rejection. Rewrite the formula or pull it out of the calc entirely.
  2. Resolve every flag within 7 days. Update the card, re-audit just that one card, export again.
  3. Replace the trust section on the live site with the auto-generated one.
  4. Save FORMULA_AUDIT_REPORT.md in the GitHub repo — it's a credibility artifact, link to it from social posts and dealer pitches.
  5. Re-run the audit quarterly. Source numbers drift (KASKO rates, fuel prices, tax tables). Stale audit = stale trust.

Why this is the right first thing

You were going to launch with hand-written trust copy and hope a dealer or journalist didn't probe too hard. Now: the audit happens once (a few focused hours), the artifact is permanent, the trust section auto-generates from it, every later dealer pitch is anchored in it, and the same tool audits your next calculator with zero rewrite. One file, three months of saved face.
